Doubow v1 execution plan — four phases from segment definition to enrollment

1
Define the v1 segment Before any code

Pick one geography + role family

e.g. "remote-global tech roles" or "West Africa + finance/ops". One segment means one adapter, one coverage story, one honest UI label.

Confirm the user persona

Early-career diaspora pro looking remote? Mid-career local hire? This sets what the re-ranker should weight — recency, sponsorship, seniority.

Decide on coarse location capture

Ask users for country at onboarding. No GPS needed. Used for coverage transparency ("we have strong listings for remote + US/EU") and re-rank weighting.

Gate: segment doc exists before sprint 1 starts
2
Ship one adapter end-to-end Sprint 1 — ~1 week

Implement ProviderAdapter base class

Fields: fetch() → raw, to_canonical(raw) → JobIn, source_label string. Keeps all adapters testable in isolation.

Wire Adzuna adapter first

Free API key, no ToS ambiguity, good for remote + global roles. Canonical output: title · company · location · apply_url · external_id · posted_at · source="adzuna".

Add fingerprint dedup to the Celery task

hash(title + company + location) stored in Redis with 48h TTL. Runs before embed to avoid wasting API calls on duplicates.

Embed and write to pgvector

Embed f"{title} {company} {description[:500]}" via text-embedding-3-small. Store vector + metadata. Pin the model now — never change without full re-index.

Surface source attribution in API + UI

Every job card shows "via Adzuna" + link to original posting. Non-negotiable for trust and ToS compliance.

Gate: 500+ indexed jobs, one end-to-end match test passing
3
Prefill the index + test matching Sprint 2 — before marketing

Run ingestion on a schedule for 2 weeks

Daily Celery beat task. Target 2,000–5,000 indexed jobs before first user. Empty index = terrible first impression.

Test with 10 real resumes from your team

Parse → embed → query → inspect top 20 results manually. Tune re-rank weights (location, seniority, sponsorship flag) based on what looks wrong.

Measure coverage honestly

If <30% of test resumes get 10+ relevant matches → either the segment is too narrow, the adapter is wrong, or re-rank is mis-weighting. Fix before enrolling users.

Add job freshness expiry

Mark jobs older than 30 days as stale=true. Exclude from default queries. Show freshness date on every card.

Gate: 10/10 internal resumes return ≥10 relevant matches
4
Widen only after the loop is stable Sprint 3+ — only when gates 1–3 pass

Add RemoteOK adapter

Public JSON feed, explicit ToS allowance. Good remote coverage. Same adapter interface — drops in with ~50 lines.

Add SerpApi as a supplemental layer

Triggered by resume keywords + location, not a blanket crawl. Set a hard monthly query budget. Treat as enrichment, not foundation.

Evaluate one commercial aggregator

Run a TheirStack or Coresignal trial only after you know which gaps the free adapters leave. Don't pay for breadth you can't match yet.

Gate: real user match rate measured, gaps identified, budget approved
